Multiplexing and Demultiplexing

Posted on 2021-02-15


这篇博文做的是传输层的准备工作


我们需要知道的是网络层，跑的就是IP协议，网络层负责实现从host到host的传输，做的事情就是寻址、路由之类，它负责连通两个host，但是连通之后怎么管的事，由上一层传输层来处理。我们可以想象网络层的作用就是在两个host之间连上了一根大的光缆(实际上是不同的，但这里可以用等效的思想)，这根光缆里面有许多根互相独立的光纤，各种比特在里面跑。相当于在两个host之间架设好了一根大的数据管道


而传输层的任务就是用好这根数据管道，将光缆资源一根根分开，落实到每一个进程上，建立相互独立的进程之间的网络连接，这就叫做 transport-layer multiplexing and demultiplexing 。
其实从后面的内容可以知道，建立process to process的连接，依靠IP地址和端口号就可以实现了。另一方面，我们也可以看到后面TCP的流量控制其实是基于线程之间的，换句话说，如果当前两个host之间大部分是UDP，那么TCP的那部分流量控制起到的效果是比较小的。再有，通过与实体光纤的联想，TCP支持全双工而HTTP支持半双工也就可以理解了。因为TCP实现全双工的本质就是两组数据在两根独立的光纤上面跑的

Multiplexing and Demultiplexing


一个直接的任务就是当多个传输层的segments传来之后，各个进程要想办法去认领属于自家进程的segment


实现的方法就是数据并不是直接传给process，而是每个process可以接多个sockets 每个socket都有属于自己的独立的标识号。而传输层就检查segment的报文段，将对应的segment移交给对应的socket。这个过程叫做demultiplexing


而在host端，给不同的sockets的数据打包装成segment，提供报文段，再把来自多个进程的segments一起交给网络层的过程，叫做multiplexing

Port Number Field


实现多路复用的关键在于使用一种机制，将每个segment引导到对应的socket。使用的方法就是 port number:


由两块组成，source port number field 和 destination port number field，各自占2个字节，一共就是4个字节。由它们来组合判断对应的socket。每个field的端口号范围为0到65535，而且0到1023的端口号是well-known port numbers，意思就是说它们被常见的应用层协议所预定了，比如HTTP预定端口号为80，FTP预定端口号为21。这里就看的很明显了，端口号是由应用层去指定的，可手动指定也可以自动分配


UDP和TCP使用端口的方式有些不同，下面先讲UDP

UDP and Multiplexing


而具体的实现过程为：每个socket被分配1个端口号，host接收到segment后，检查segment报文里面的destination port number，然后将其引导到有相同端口号的socket上


需要注意的是，在UDP multiplexing的过程中，没有涉及到IP地址的问题，涉及到的只是两个端口号，以及对应端口号的两个sockets


在UDP里面，如果两个segment有不同的source IP addresses但是有相同的destination IP和destination port number，那么它们会被引导到同一个socket上去


虽然在确定把segment引导到哪个socket只需要destination port number，但是resource port number相当于是return address，当然实际的回传需要source IP和source port number一起来进行定位。IP地址负责送到host，port number 负责送到socket


本质就是UDP只看destination port number，号对了就引导给socket
